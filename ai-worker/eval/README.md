# Evaluation harness

A lightweight, local-first eval harness for the ai-worker capabilities.
Kept deliberately separate from `app/` so the production pipeline never
imports eval code and vice versa.

## What this is (and is not)

**Is:** a handful of pure-Python metrics, JSONL-in / JSON+CSV-out
report writers, and two harness functions (`run_rag_eval`,
`run_ocr_eval`) that one developer can iterate on from the command
line without any tracking platform, cloud service, or model server.

**Is not:** a replacement for a real model-quality pipeline. There is
no experiment database, no leaderboard, no hyperparameter sweep
driver, no CI gating. When the project outgrows flat files, swap
them out — nothing in `app/` depends on anything here.

## Dataset schemas

### A. Text RAG eval — `eval/datasets/rag_sample.jsonl`

One JSON object per line. Lines starting with `#` and blank lines are
skipped so you can keep inline comments in the file.

| Field               | Type           | Required | Notes                                                                 |
|---------------------|----------------|----------|-----------------------------------------------------------------------|
| `query`             | string         | yes      | The user query to test.                                               |
| `expected_doc_ids`  | list<string>   | no       | Doc ids we expect in top-k. Rows without this field are excluded from hit@k and MRR aggregation. |
| `expected_keywords` | list<string>   | no       | Substrings we expect in the generated answer (case-insensitive match). |
| `notes`             | string         | no       | Free-form annotation for the human author. Not scored.                |

Example:

```jsonl
{"query": "who runs the bookshop at the end of the railway line?", "expected_doc_ids": ["anime-003"], "expected_keywords": ["bookshop", "translator"], "notes": "cozy mystery"}
```

### B. OCR eval — `eval/datasets/ocr_sample.jsonl`

| Field          | Type    | Required | Notes                                                                                 |
|----------------|---------|----------|---------------------------------------------------------------------------------------|
| `file`         | string  | yes      | Path **relative to this JSONL file's directory**. Supported: `.png`, `.jpg`, `.jpeg`, `.pdf`. |
| `ground_truth` | string  | yes      | Exact expected extracted text (UTF-8). Whitespace is normalized in the CER/WER pass.  |
| `language`     | string  | no       | Tesseract language code — `eng`, `eng+kor`, `kor`, `jpn`, `chi_sim`, etc. For CJK codes the harness reports WER as `None` and excludes the row from WER aggregation (whitespace-split WER is not meaningful there). |
| `notes`        | string  | no       | Free-form annotation.                                                                 |

Example:

```jsonl
{"file": "samples/hello_world.png", "ground_truth": "HELLO WORLD", "language": "eng", "notes": "bare minimum smoke test"}
```

### C. Multimodal eval — `eval/datasets/multimodal_sample.jsonl` (schema only, no harness yet)

| Field               | Type         | Required | Notes                                                                      |
|---------------------|--------------|----------|----------------------------------------------------------------------------|
| `image`             | string       | yes      | Path relative to the JSONL file (same rule as OCR).                        |
| `question`          | string       | yes      | Natural-language question about the image.                                 |
| `expected_answer`   | string       | no       | Canonical short answer for exact / substring match.                        |
| `expected_keywords` | list<string> | no       | Substrings the answer must contain.                                        |
| `expected_labels`   | list<string> | no       | Visual labels the model should identify (object names, colors, etc.).      |
| `requires_ocr`      | bool         | no       | True if the task demands text extraction before answering.                 |
| `language`          | string       | no       | OCR/VLM language code when `requires_ocr` is true.                         |
| `notes`             | string       | no       | Free-form annotation.                                                      |

**Status: placeholder only.** There is no multimodal harness yet. The
schema is committed so a future phase can wire a scorer against it
without another round of dataset design. Recommended future metrics
(all reusable from `eval.harness.metrics`): exact match, substring
match, `keyword_coverage`, label recall/precision, and CER over any
VLM-reported OCR subfield.

## Metrics (what each one means)

- **`hit@k`** — 1.0 if ANY `expected_doc_id` appears in the retriever's
  top-k, else 0.0. `None` when the row has no `expected_doc_ids`. Simple
  binary gate on "did we recall something useful".
- **`reciprocal_rank` / `MRR`** — 1/rank of the first matching
  expected id in the ranked list, 0.0 if none appears, `None` for
  rows without `expected_doc_ids`. MRR is the mean over non-None
  rows. Penalizes burying the right answer at position 5 vs position 1.
- **`keyword_coverage`** — fraction of `expected_keywords` present as
  substrings (case-insensitive) in the generated answer. Cheapest
  reasonable signal for "the generator actually mentioned the thing".
- **`CER`** — character edit distance / reference character count,
  after whitespace normalization. 0.0 is perfect. Above ~0.1 usually
  means noticeably degraded OCR; above ~0.3 means the output is not
  usable for downstream retrieval.
- **`WER`** — same thing at whitespace-split word level. Not
  meaningful for CJK languages — the harness reports `None` for those.
- **`empty_rate`** — fraction of rows where the OCR engine produced
  zero (normalized) characters. High empty_rate usually means a
  misconfigured language pack or unreadable scans.
- **latency (ms)** — wall-clock for each provider call. p50, mean,
  and max are reported. Useful for regression-checking new models.

## Recommended evaluation sequence

Iterate in this order — each stage's metric gates the next:

### 1. Text RAG baseline

**Goal:** confirm the retriever reaches "mostly right docs, mostly
right order" before touching OCR at all.

**Run:**
```bash
cd ai-worker
python -m scripts.build_rag_index --fixture     # one-time
python -m eval.run_eval rag \
  --dataset eval/datasets/rag_sample.jsonl \
  --top-k 5
```

**Gate:** `mean_hit_at_k ≥ 0.80` **and** `MRR ≥ 0.50` on the fixture.
If you fall below, the problem is in embeddings/chunking/config — do
NOT move on to OCR yet. Raise `top_k` (`--top-k 10`) only as a
diagnostic, not as a fix.

### 2. OCR extraction quality

**Goal:** confirm the OCR provider produces text that is *clean
enough to serve as retrieval input*. This is a prerequisite for the
combined stage.

**Run:**
```bash
python -m scripts.make_ocr_sample_fixtures       # one-time
python -m eval.run_eval ocr \
  --dataset eval/datasets/ocr_sample.jsonl
```

**Gate:** `mean_cer ≤ 0.10` and `empty_rate == 0.0` on the sample
fixtures. On real curated scans, `mean_cer ≤ 0.20` is usually the
threshold where RAG downstream still behaves sanely.

If CER is above 0.25, don't proceed — debug the provider first. Common
causes: missing language pack, too low `ocr_pdf_dpi`, missing
preprocessing on low-contrast scans.

### 3. OCR + RAG combined (future, not automated yet)

**Goal:** end-to-end test that OCR output is actually usable as RAG
input on real documents.

**When you get there:** build a small combined dataset of
`{file, ground_truth_text, query, expected_doc_ids}`, run the OCR
harness to produce extracted text, feed the extracted text into the
RAG harness as `query`, and compare the two hit@k numbers:

- RAG-on-ground-truth: upper bound, treats OCR as perfect.
- RAG-on-OCR-output: real-world behavior.

The delta is what OCR is costing you. **Gate:** the two numbers should
be within 0.10 of each other; anything worse means OCR quality is
blocking retrieval on real inputs. **This stage is currently manual**
— see "What is still manual" at the bottom of this doc.

### 4. Multimodal

**Goal:** confirm the full multimodal pipeline (OCR + vision + fusion +
retrieval + generation) produces answers that mention the right keywords
and surface the correct visual labels.

**Run:**
```bash
python -m scripts.make_multimodal_sample_fixtures    # one-time
python -m eval.run_eval multimodal \
  --dataset eval/datasets/multimodal_sample.jsonl
```

**Gate:** `mean_keyword_coverage >= 0.60` **and**
`mean_substring_match >= 0.50` on the fixture. If you fall below,
the problem is likely in the vision provider or the fusion step —
check the MULTIMODAL_TRACE artifacts for stage-level diagnostics.

**Filtering:** use `--require-ocr-only` to evaluate only OCR-dependent
rows. Use `--vision-provider heuristic|claude` to override the vision
provider for A/B comparison.

**Stage-level latency breakdown:** the harness reports OCR, vision,
and retrieval+generation latency separately when `emit_trace=True`
(enabled automatically during eval runs). Use these to identify
which stage is the bottleneck.

## `retrieval` mode — dense-retrieval baseline measurement

Separate from the `rag` mode (which scores generator output too), the
`retrieval` mode measures **only** the dense-retrieval stage. No
generator is invoked, and the output is four per-run artifacts:

```
eval/reports/retrieval-<timestamp>/
├── retrieval_eval_report.json     summary + per-row metrics
├── retrieval_eval_report.md       human-readable summary
├── top_k_dump.jsonl               one record per (query, rank) pair
└── duplicate_analysis.json        per-query + aggregate dup stats
```

**Per-row metrics**: hit@1, hit@3, hit@5, mrr@10, ndcg@10 (binary
relevance), dup_rate, unique_doc_coverage, top1_score_margin,
avg_context_token_count, expected_keyword_match_rate.

**Run** against the offline anime corpus (no Postgres needed):

```bash
cd ai-worker

# 1. Stage the corpus (one-time, see eval/corpora/anime_namu_v3/README.md)
cp 'D:/port/rag/app/scripts/namu_anime_v3.jsonl' \
   eval/corpora/anime_namu_v3/corpus.jsonl

# 2. Smoke test on 6 hand-authored queries
python -m eval.run_eval retrieval \
    --corpus  eval/corpora/anime_namu_v3/corpus.jsonl \
    --dataset eval/eval_queries/anime_smoke_6.jsonl \
    --top-k 10

# 3. Full silver baseline (200 deterministic synthetic queries)
python -m eval.run_eval retrieval \
    --corpus  eval/corpora/anime_namu_v3/corpus.jsonl \
    --dataset eval/eval_queries/anime_silver_200.jsonl \
    --top-k 10

# 4. Gold baseline (20 hand-curated queries — trust these as ground truth)
python -m eval.run_eval retrieval \
    --corpus  eval/corpora/anime_namu_v3/corpus.jsonl \
    --dataset eval/eval_queries/anime_gold_20.jsonl \
    --top-k 10
```

The `retrieval` mode also accepts the live ragmeta/FAISS path (omit
`--corpus`) — same behavior as the legacy `rag` mode without the
`--offline-corpus` flag.

See [`eval_queries/README.md`](eval_queries/README.md) for the silver
vs. gold semantics, the deterministic generator's stratification, and
the gold-set extension recipe. See
[`corpora/anime_namu_v3/README.md`](corpora/anime_namu_v3/README.md)
for the corpus schema and re-stage instructions.

### Phase 0 baseline tooling (post-retrieval)

Three companion subcommands run against an existing retrieval run.
None of them re-embeds, so they're cheap.

```bash
# Add doc/keyword cross-tab to a retrieval run that didn't emit one
python -m eval.run_eval retrieval-miss-analysis \
    --report-dir eval/reports/retrieval-silver200-baseline \
    --top-k 10

# Side-by-side compare two retrieval runs (deterministic vs opus).
# Auto-emits a Caveats block + per-slice retriever_config in the .md.
python -m eval.run_eval retrieval-compare \
    --deterministic-report eval/reports/retrieval-silver200-baseline/retrieval_eval_report.json \
    --opus-report          eval/reports/retrieval-silver200-opus-baseline/retrieval_eval_report.json \
    --deterministic-max-seq-length 8192 \
    --opus-max-seq-length 1024 \
    --out-json eval/reports/retrieval-baseline-comparison.json \
    --out-md   eval/reports/retrieval-baseline-comparison.md

# Same compare, but with a hyperparameter-tuned variant on the
# deterministic side. The tuned slice (and its diagnostic) is rendered
# in its own headline-metrics table so it never shares a row with
# baseline numbers.
python -m eval.run_eval retrieval-compare \
    --deterministic-report eval/reports/retrieval-silver200-tuned/retrieval_eval_report.json \
    --opus-report          eval/reports/retrieval-silver200-opus-baseline/retrieval_eval_report.json \
    --deterministic-kind tuned \
    --opus-kind baseline \
    --out-json eval/reports/retrieval-tuned-vs-baseline.json \
    --out-md   eval/reports/retrieval-tuned-vs-baseline.md

# Tokenizer-based char/token length distribution for the corpus.
# Use this to size max_seq_length caps with measured numbers instead
# of char-derived guesses.
python -m eval.run_eval analyze-corpus-lengths \
    --corpus eval/corpora/anime_namu_v3/corpus.jsonl \
    --tokenizer BAAI/bge-m3 \
    --out-json eval/reports/corpus-length-analysis.json \
    --out-md   eval/reports/corpus-length-analysis.md
```

The full Phase 0 trade-off log lives in
[`reports/phase0-baseline-tradeoffs.md`](reports/phase0-baseline-tradeoffs.md).

## Running the eval CLI

Both subcommands print a short human summary to stdout and write a
JSON report (and, by default, a CSV). Reports go to
`eval/reports/{mode}-{timestamp}.{json,csv}` unless `--out-json` /
`--out-csv` are passed.

```bash
# Text RAG — builds the real production stack (bge-m3 + FAISS + ragmeta)
python -m eval.run_eval rag \
    --dataset eval/datasets/rag_sample.jsonl \
    --out-json eval/reports/rag-latest.json \
    --out-csv  eval/reports/rag-latest.csv \
    --top-k 5

# OCR — builds the real Tesseract + PyMuPDF provider
python -m eval.run_eval ocr \
    --dataset eval/datasets/ocr_sample.jsonl \
    --out-json eval/reports/ocr-latest.json \
    --out-csv  eval/reports/ocr-latest.csv

# Multimodal — builds the full MULTIMODAL capability (OCR + vision + RAG)
python -m eval.run_eval multimodal \
    --dataset eval/datasets/multimodal_sample.jsonl \
    --out-json eval/reports/multimodal-latest.json \
    --out-csv  eval/reports/multimodal-latest.csv

# Multimodal — OCR-only rows with Claude vision provider
python -m eval.run_eval multimodal \
    --dataset eval/datasets/multimodal_sample.jsonl \
    --require-ocr-only \
    --vision-provider claude
```

CLI flags:

| Flag                | Applies to | Default                                                   |
|---------------------|------------|-----------------------------------------------------------|
| `--dataset PATH`    | both       | (required)                                                |
| `--out-json PATH`   | both       | `eval/reports/{mode}-<timestamp>.json`                    |
| `--out-csv PATH`    | both       | `eval/reports/{mode}-<timestamp>.csv`                     |
| `--no-csv`          | both       | emit JSON only                                            |
| `-v` / `--verbose`  | both       | DEBUG logging                                             |
| `--top-k N`         | rag        | worker's `AIPIPELINE_WORKER_RAG_TOP_K` (default `5`)      |
| `--fail-missing`    | ocr        | treat missing fixture files as errors instead of skipping |

## Programmatic use

The harness is importable for unit tests or custom runners:

```python
from eval.harness import (
    run_rag_eval, run_ocr_eval,
    cer, wer, hit_at_k, reciprocal_rank, keyword_coverage,
)
```

Both `run_rag_eval` and `run_ocr_eval` take already-constructed
retriever/generator/provider objects — no config coupling — so tests
can hand in fakes and a custom runner can hand in a GPU-backed
provider without touching this package.

## Ignore rules / `.gitignore`

Reports are generated and should not be committed. If you use git
locally, add:

```
ai-worker/eval/reports/
ai-worker/eval/datasets/samples/
```

The `samples/` folder holds synthesized OCR fixture images produced
by `scripts/make_ocr_sample_fixtures.py`; regenerating them is cheap
and they drift across fonts.

## What is still manual / not automated yet

These are deliberate phase-scope lines, not oversights:

1. **Dataset curation.** You write the JSONL files by hand. No scraping,
   no LLM-generated questions. Small curated sets beat large noisy ones
   for an iteration loop of one developer.
2. **OCR → RAG chaining eval.** There is no harness that takes OCR
   output as RAG input. For now, run the OCR harness, copy an
   OCR_TEXT artifact into a new RAG dataset's `query` field, re-run
   the RAG harness, and diff by eye against the ground-truth RAG run.
   The architecture supports this flow — see
   [architecture.md](../../docs/architecture.md) — but the automation
   belongs in a later phase.
3. **Multimodal scoring.** Schema committed, harness deferred.
4. **Regression gating in CI.** The harness exits 0 on a successful
   run regardless of metric quality. When you want CI gating, wrap
   the CLI in a post-run check that reads the JSON report and fails
   if e.g. `mean_cer > 0.15`. Don't bake thresholds into the harness
   itself — they should live alongside the project's release criteria,
   not in the tool.
5. **Latency baselining across machines.** Latencies in the report are
   raw wall-clock numbers. They're only comparable within one
   machine / one run. Cross-machine comparison needs a fixed-hardware
   harness, which is out of scope here.
