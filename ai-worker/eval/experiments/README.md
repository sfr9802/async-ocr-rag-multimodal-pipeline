# eval/experiments — Optuna round-refinement workspace

This directory holds the project-side artifacts for round-based Optuna
tuning. The Optuna orchestration itself (sampler / pruner construction,
trial dispatch, bundle export, schema validation, markdown rendering) is
owned by the installed `optuna-round-refinement` skill — see
`C:\Users\sfr99\.claude\skills\optuna-round-refinement\SKILL.md`.

The project-side contributes only four kinds of artifact:

1. **An `evaluate(params: dict) -> dict` callable** at
   [`ai-worker/eval/tune_eval.py`](../tune_eval.py), referenced from
   each round config via `evaluate: "eval.tune_eval:evaluate"`.
2. **Round configs** under `rounds/round_NN_config.json`, validated
   against the skill's `next_round_config.schema.json`.
3. **Round analyses** under `rounds/round_NN_analysis.md` (one per
   completed round, produced by the `/analyze-study` flow).
4. **Study receipts** at the top of each study directory:
   `FINAL_BEST.json` and `STUDY_SUMMARY.md`.

## Layout

```

│   ├── round_01_analysis.md
│   ├── round_02_config.json
│   └── ...
├── run_output/              ← gitignored: per-run skill outputs
│   ├── .gitkeep
│   ├── study_bundle.json
│   └── llm_input.md
└── studies/                 ← legacy scripts/tune.py study artefacts (kept for replay)
```

## Round cadence

- **One round per git branch**, or at minimum **one round per commit**.
  The round_NN_config.json is the input; the analysis markdown + the
  next round_(NN+1)_config.json are the output.
- Bundles (`study_bundle.json`) and the rendered `llm_input.md` live
  under `run_output/` and are gitignored — they're regenerated by the
  skill on every run.
- Configs (`rounds/round_NN_config.json`) are committed so any reviewer
  can reproduce the exact search space and provenance hashes used to
  drive the round.
- Analysis markdowns (`rounds/round_NN_analysis.md`) and study receipts
  (`FINAL_BEST.json`, `STUDY_SUMMARY.md`) are committed so the
  round-to-round narrative survives without needing the on-disk SQLite
  study DB.

## Driving a round

From `ai-worker/`:

```bash
python "C:/Users/sfr99/.claude/skills/optuna-round-refinement/scripts/round_runner.py" run \
    --config eval/experiments/rounds/round_01_config.json \
    --out-bundle eval/experiments/run_output/study_bundle.json \
    --out-llm-input eval/experiments/run_output/llm_input.md \
    --evaluate-search-path .
```

The runner imports `eval.tune_eval:evaluate`, drives `n_trials` Optuna
trials against the shared retriever bundle, writes the schema-validated
bundle and rendered LLM input, then hands control back. Round analysis
and the next round's config are produced by the skill's analyst prompts
(`/analyze-study`, `/propose-next`) operating on the bundle alone.

## What this directory does NOT contain

- Optuna sampler / pruner / suggest_* code — owned by the skill.
- Schema files — owned by the skill (`schemas/next_round_config.schema.json`,
  `schemas/study_bundle.schema.json`).
- Templates — owned by the skill (`templates/llm_input.md`,
  `templates/round_report.md`).
- Eval harness internals — see [`ai-worker/eval/harness/`](../harness/).
